================================================================================
  ⚠️  ตารางใน DB ที่ดูเหมือนซ้ำซ้อน / ต้องตรวจสอบ
================================================================================

  ISSUE 1: DWG Request มี 2 ระบบ
  ─────────────────────────────────────────────────────────────────────────────
  tool_dwg_request   ← ระบบเก่า (ใช้ใน eng_mtc_model.js / Tooling Inspect)
  tr_request         ← ระบบใหม่ (General DWG Request)
  tr_workflow        ← ระบบใหม่ (workflow history)

  → tr_request มี 35 legacy records ที่ migrate มาจากระบบเก่า
    (request_item = NULL หมายถึง record เก่า)
  → แนะนำ: ตัดสินใจว่าจะ deprecate tool_dwg_request หรือไม่

  ISSUE 2: Workflow tables อาจซ้ำกับ tr_workflow
  ─────────────────────────────────────────────────────────────────────────────
  approval           ← ใช้ใน sds.js (approval ของ setup sheet)
  eng_check          ← มีใน DB แต่ไม่พบการใช้ใน mtc files
  draft_man          ← มีใน DB แต่ไม่พบการใช้ใน mtc files
  dwg_check          ← มีใน DB แต่ไม่พบการใช้ใน mtc files
  eng_review         ← มีใน DB แต่ไม่พบการใช้ใน mtc files
  eng_approve        ← มีใน DB แต่ไม่พบการใช้ใน mtc files
  eng_inform         ← มีใน DB แต่ไม่พบการใช้ใน mtc files

  → tables เหล่านี้ชื่อตรงกับ stage ของ General DWG Request
  → อาจเป็น draft schema เก่าก่อนจะรวมเป็น tr_workflow
  → แนะนำ: ตรวจสอบว่ามีข้อมูลใน tables เหล่านี้หรือไม่ ถ้าว่างสามารถ DROP ได้

  ISSUE 3: tooling master ซ้ำซ้อน
  ─────────────────────────────────────────────────────────────────────────────
  tooling            ← generic tooling table (มีใน DB แต่ไม่พบการใช้ใน mtc files)
  tooling_ksb22g     ← machine-specific tooling
  tooling_ksb80      ← machine-specific tooling
  ... (อีก 7 tables)

  → tooling ทั่วไปกับ tooling_* เฉพาะเครื่องอาจซ้ำกัน
  → แนะนำ: ตรวจสอบโครงสร้าง column ว่าต่างกันอย่างไร
