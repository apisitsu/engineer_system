# รายงานการตรวจสอบระบบจัดการสิทธิ์การเข้าถึง (Permissions Audit Report)

## 1. ขอบเขตความรับผิดชอบและการตรวจสอบ (Audit Scope)
ดำเนินการตรวจสอบโครงสร้างและตรรกะของระบบ Permissions ของ Kanban Module ทั้งในฝั่ง Backend (`kanban_acl.js`, `kanban_card.js`, `kanban_board.js`) และ Frontend (`useKanbanPermissions.js`) เพื่อประเมินความปลอดภัย ความถูกต้อง และความซับซ้อนของโครงสร้างสิทธิ์การใช้งาน

## 2. ผลการตรวจสอบและประเมินระบบ (Findings & Assessment)

### จุดแข็ง (Strengths)
- **Centralized ACL:** ระบบมีการรวมตรรกะการตรวจสอบสิทธิ์ไว้ที่ไฟล์เดียว (`kanban_acl.js`) ทำให้การบำรุงรักษาและการตรวจสอบทำได้ง่ายและเป็นระบบ
- **Role Hierarchy:** การแบ่งลำดับชั้นของสิทธิ์ (Global -> Project -> Board -> Card) มีความชัดเจน เข้าใจง่าย และปรับแต่งได้สะดวก
- **Super Admin (God Mode):** สิทธิ์ `AD` ทำงานได้อย่างถูกต้องและถูกแยกออกจากบทบาทของ `MGR / COORD` อย่างชัดเจน ช่วยให้การดูแลระบบเป็นไปอย่างมีประสิทธิภาพ

### จุดอ่อนและข้อบกพร่องที่พบ (Flaws & Bugs)
1. **Insecure Direct Object Reference (IDOR) ใน API ดึงข้อมูล:**
   - **`GetCard` & `GetTaskLists`:** ไม่มีการตรวจสอบสิทธิ์ก่อนว่าผู้ใช้งานมีสิทธิ์เข้าถึงบอร์ดหรือโปรเจกต์ของการ์ดใบนั้นหรือไม่ หากการ์ดนั้นเป็นสาธารณะ (`is_private = false`) ผู้ใช้ใด ๆ ที่ล็อกอินในระบบจะสามารถดึงข้อมูลการ์ดหรือรายการงาน (Task Lists) ของโปรเจกต์ที่เป็นส่วนตัวได้เพียงแค่ระบุหมายเลข ID ให้ถูกต้อง
   - **`GetLists`, `GetLabels` และ `GetBoardMembers`:** ไม่มีการตรวจสอบสิทธิ์ของ Board (ขาด `canViewBoard`) ทำให้สามารถดึงข้อมูลลิสต์และป้ายกำกับของบอร์ดใด ๆ ได้
2. **ตรรกะการตรวจสอบสิทธิ์คลาดเคลื่อน (Flawed Logic):**
   - **`canViewCard` (Backend):** คืนค่า `true` ทันทีที่การ์ดไม่ได้เป็นส่วนตัว โดยไม่ได้ตรวจสอบย้อนกลับว่าผู้ใช้งานมีสิทธิ์เข้าถึงโปรเจกต์ต้นสังกัดของการ์ดใบนั้นหรือไม่
   - **`canEditCard` (Backend):** อนุญาตให้ "สมาชิกบอร์ดทุกคน" สามารถแก้ไขการ์ดได้ (รวมถึงผู้ชม หรือ Viewer ของบอร์ด) ซึ่งถือเป็นการให้สิทธิ์เกินความจำเป็น (Over-permissive)
3. **ความไม่สอดคล้องระหว่าง Frontend และ Backend (Mismatches):**
   - **Frontend UI (`useKanbanPermissions.js`):** ไม่ได้อนุญาตให้ `Board Editor` สามารถแก้ไขการ์ดได้ แต่ระบบ Backend กลับอนุญาตให้แก้ไขได้ ทำให้เกิดปัญหา UI แสดงผลเป็น Read-only ทั้งที่ผู้ใช้งานควรมีสิทธิ์แก้ไข

## 3. แผนการพัฒนาและปรับปรุง (Development & Improvement Plan)
- **อัปเดต `kanban_acl.js`:** 
  - แก้ไขฟังก์ชัน `canEditCard` ให้เรียกใช้ `canEditBoard` เพื่อให้ครอบคลุมเฉพาะผู้ดูแลและเอดิเตอร์ (Owner/Editor) แทนการให้สิทธิ์ผู้ใช้บอร์ดทุกคน
  - แก้ไขฟังก์ชัน `canViewCard` ให้ตรวจสอบการเข้าถึงในระดับโปรเจกต์ (`canAccessProject`)
  - เพิ่มฟังก์ชัน `canViewBoard` เพื่ออุดช่องโหว่การดึงข้อมูลส่วนประกอบภายในบอร์ด
- **อุดช่องโหว่ IDOR ใน `kanban_card.js` และ `kanban_board.js`:** 
  - แทรกคำสั่งการตรวจสอบของ `canViewCard` และ `canViewBoard` ลงใน Endpoint ขา GET ทุกเส้นที่เกี่ยวข้อง
- **แก้ไข `useKanbanPermissions.js` ใน Frontend:**
  - ปลดล็อกและนำตัวแปร `canEditBoard` กลับเข้าไปรวมในตรรกะการประเมินผลของ `canEditCard`

## 4. ผลลัพธ์การแก้ไข (Execution Outcomes)
- ได้ดำเนินการปรับปรุงและแก้ไขช่องโหว่ความปลอดภัยในระบบเรียบร้อยแล้วทั้งหมด
- ฝั่ง Backend API และ Frontend UI มีความสอดคล้องกันอย่างสมบูรณ์ ปิดช่องโหว่การเข้าถึงและดึงข้อมูลโดยไม่ได้รับอนุญาต (IDOR) สำเร็จเรียบร้อย
